# OpenCode 1.14.46 Cursor Provider Visibility Fix

## Date
2026-05-10

## User Report
OpenCode UI did not show the Cursor provider. The initial suspicion was that the plugin was not loaded.

## Findings
- Global OpenCode config still declared the Cursor provider stub and local plugin path.
- The local plugin repository existed at `C:\NEW PRG\opencode-cursor` and had a valid `package.json`, `dist/index.js`, and built runtime files.
- Recent OpenCode logs showed the local plugin path being loaded, so this was not a simple missing-file or bad-path issue.
- Older OpenCode 1.14.41 logs showed `providerID=cursor found`, while current OpenCode 1.14.46 logs loaded the plugin but omitted `cursor` from the provider list.
- The likely compatibility break is that OpenCode 1.14.46 filters out custom providers before the auth loader can dynamically attach discovered models when the provider only exists as a minimal/empty config stub.

## Change Made
Updated `src/index.ts` so the plugin now registers a `config` hook. The hook ensures the `cursor` provider config contains:

- `name: "Cursor"`
- `npm: "@ai-sdk/openai-compatible"`
- a minimal `auto` model stub

This makes the provider visible during OpenCode provider initialization. After successful Cursor OAuth, the existing auth loader still discovers real Cursor models and replaces the provider model map with the dynamic model list.

The patch also removed the previous `(provider as any)` assignment and uses the SDK `Model` type for `provider.models`, keeping the code type-safe.

## Files Changed
- `src/index.ts`
- `dist/index.js` and `dist/index.d.ts` regenerated by `bun run build`
